# KM Topic Research

## 何时使用

当用户要你做以下任务时使用本技能：
1. 给一个或多个 KM 链接（`/space/*`、`/collabpage/*` 等）并指定一个调研主题。
2. 需要从文档树与关联链接快速摸清产品/服务全景。
3. 关注产品定位、核心功能、用户场景、OpenAPI/客户端/命令行能力概览。
4. 明确要求控制阅读深度与次数，不做参数细节或调用细节研究。

当用户只要改写现有文档文本、或只分析单篇内容时，不使用本技能。

## 输入约定

开始执行前，先明确并复述以下输入：
1. `topic`：本次调研主题（例：S3 产品与开放能力）。
2. `seed_docs`：种子文档 URL 列表。
3. `depth`：默认 `overview`（可选 `overview|standard`）。
4. `scope_hint`：可选的关注范围（例：只看 OpenAPI 与 CLI）。

默认值：
1. `depth=overview`
2. 若用户未指定，默认“优先业务全景，不下钻参数”。

## 阅读预算与节奏控制（强制）

为防止“过读”，按预算执行：
1. 每轮最多展开 5 篇新增文档。
2. 默认总阅读上限 25 篇；超限前必须先向用户汇报并确认是否继续。
3. 搜索调用默认不超过 3 次，仅用于补全缺失服务或交叉验证名称。
4. 单篇文档只抽取与主题直接相关段落，不做全篇逐段解析。
5. 遇到 API 细节内容，仅记录“能力存在与用途”，不整理参数表。

## 执行流程

1. 建立候选池：
从 `seed_docs` 提取目录（catalog）、子文档、文内跳转链接，形成第一批候选文档。

2. 轻量扩展：
在候选池不足以覆盖主题时，再使用搜索补齐明显缺口（产品别名、上游导航页、官方总览页）。

3. 主题抽取：
对每个有效服务提取固定字段：
`产品定位`、`核心功能`、`目标用户/场景`、`开放能力`（OpenAPI/客户端/CLI/SDK 是否存在）。

4. 分类与比较：
至少给出一版业务分组（例如：协作、存储、消息、组织管理、开发者能力），并在组内做简要比较。

5. 缺口标注：
无法确认的信息统一标注为“未在本轮样本中确认”，禁止主观补全。

## 输出格式（强制）

产出应包含以下部分：
1. `宏观结构`：先给全景分类图（业务域 -> 服务）。
2. `服务清单`：每个服务 4-6 行概要（定位/功能/场景/开放能力）。
3. `对比摘要`：按业务域横向比较相同点与差异点。
4. `OpenAPI/CLI 观察`：只写能力级结论，不下钻参数。
5. `阅读清单`：逐条列出已读文档（标题 + URL + 一句话用途）。
6. `边界与未覆盖`：明确哪些没查或仅做了初步确认。

## 报告风格约束

1. 先结论后证据，避免长段背景铺垫。
2. 使用业务语言，不堆接口字段名。
3. 对“推断”与“事实”做显式区分。
4. 链接可追溯：关键结论后应能对应到阅读清单中的来源。

## 触发短语示例

1. “从这个 space 文档往下查，做某主题概要调研。”
2. “只看目录、子文档和文内链接，控制阅读量，给分类比较。”
3. “补一点搜索，但不要深挖 API 参数，输出阅读清单。”

## 禁止事项

1. 未经用户确认，禁止无限扩展阅读范围。
2. 禁止在本技能里输出“接口参数详解”或“逐接口调用教程”。
3. 禁止遗漏阅读清单。
